# 리니어 크로스-어텐션 아키텍처 업데이트

**날짜**: 2026-02-26  
**작성자**: Antigravity

## 1. 개요
초기 `base-model-2` 아키텍처에서는 인코더와 디코더 간의 상호작용을 크로스-어텐션(Cross-Attention) 없이 출력 시퀀스를 단순히 이어붙이는(Concatenate) 방식으로 처리했습니다:
```python
x = torch.cat([encoder_out, tgt_emb], dim=1) # (B, src_len + tgt_len, d_model)
```
이 방식은 구현이 단순하지만 모델을 스케일링할 때 심각한 성능 병목 현상을 일으켰습니다. 이를 해결하기 위해 **속도가 매우 빠른 O(N) 리니어 크로스-어텐션 모듈**로 대체하였고, 결과적으로 VRAM 메모리를 대폭 절약하고 학습 속도를 크게 향상시킬 수 있었습니다.

---

## 2. 왜 메모리 사용량이 감소했는가?

단순 이어붙이기 방식에서 명시적인 리니어 크로스-어텐션 방식으로 변경한 후, 128M 모델 기준 VRAM 사용량이 **30GB에서 22GB로 감소**하고 학습 속도가 빨라지는 결과를 확인했습니다.

어텐션 블록이라는 새로운 파라미터를 추가했음에도 메모리가 8GB나 줄어든 것은 이례적으로 보일 수 있습니다. 이는 우리가 가장 큰 문제였던 **시퀀스 이어붙이기 병목(Sequence Concatenation Bottleneck)** 현상을 제거했기 때문입니다.

### A. Mamba 내부의 시퀀스 길이 감소
Mamba의 병렬 스캔(Parallel scan) 연산은 처리하는 시퀀스 길이에 비례하여 비용이 증가합니다.
* **이전 (이어붙이기):** 디코더의 모든 개별 레이어가 $N_{src} + N_{tgt}$ 길이의 시퀀스를 스캔해야 했습니다.
* **현재 (리니어 크로스-어텐션):** 디코더의 Mamba 블록은 오직 타겟 시퀀스인 $N_{tgt}$ 길이만 스캔합니다. 인코더의 정보($N_{src}$)는 리니어 어텐션 메커니즘을 통해 별도로 주입됩니다.

### B. 활성화(Activation) 메모리의 극적인 감소
딥러닝 모델 학습 중 차지하는 메모리의 대부분은 모델 파라미터가 아닌 **활성화 메모리(역전파를 위해 저장해 두는 중간 계산 텐서들)**입니다.
활성화 메모리 크기는 `배치 크기(Batch Size) * 시퀀스 길이(Sequence Length) * 은닉 차원(Hidden Dimension)`에 정비례합니다.

10층(10-layer) 깊이의 디코더 내부에서 효과적인 시퀀스 길이를 대략 절반으로 줄임으로써(Mamba 정방향 연산에서 $N_{src}$를 제거), 역전파 연산을 위해 필요한 메모리가 기하급수적으로 줄어들어 기가바이트(GB) 단위의 VRAM을 절약할 수 있었습니다.

---

## 3. 왜 학습 속도가 빨라졌는가?

1. **더 짧은 스캔 연산**: Mamba의 하드웨어 친화적인 선택적 스캔(Selective scan)은 매우 빠르지만, 여전히 시퀀스 길이에 비례해 시간이 소모됩니다. 10개 레이어에 걸쳐 $N_{src} + N_{tgt}$ 토큰을 반복적으로 스캔하는 것보다 $N_{tgt}$ 토큰만 스캔하는 것이 본질적으로 훨씬 빠릅니다.
2. **O(N) 리니어 어텐션의 효율성**: 
   일반적인 Softmax 크로스-어텐션 방식이었다면 $O(N_{src} \times N_{tgt})$ 연산이 요구되어 모델 속도가 크게 느려졌을 것입니다. 
   하지만 이번에 구현한 **리니어 크로스-어텐션**은 전체 인코더 시퀀스를 $K^T V$ 연산을 통해 고정된 크기의 $KV$ 문맥 행렬(Context Matrix)로 압축합니다. 이는 Tensor Core가 거의 즉시 계산할 수 있는 순수 행렬 곱셈(Matrix Multiplication)이므로 완벽한 O(N) 스케일링 곡선을 제공합니다.

---

## 4. 문맥 행렬(Context Matrix) 용량 문제 및 해결책

디코더가 4096개 토큰에 달하는 긴 시퀀스를 처리할 때, 모든 $N_{src}$ 토큰을 단일 문맥 행렬 하나에 압축하면 심각한 정보 병목(파괴적 중첩 현상, Catastrophic superposition)이 발생할 수 있습니다.

초기 구현에서는 극단적인 계산 속도를 위해 단일 $KV$ 헤드를 사용하는 **MQA (Multi-Query Attention)** 방식을 채택했었습니다. 
* 8M 모델 기준 전체 정보 저장 용량: `72 x 72 = 5,184개 요소`.
* 이 용량은 Mamba 내부의 SSM 상태 크기 $(576 \times 16 = 9,216)$ 보다도 적었으며, 4096개의 개별 토큰 문맥을 정확하게 저장하기엔 턱없이 부족했습니다.

**해결 방안:**
이러한 문제를 해결하기 위해 `decoder.py`의 리니어 크로스-어텐션 구성을 **MQA**에서 **MHA (Multi-Head Attention)**로 전환했습니다.
* `n_kv_heads`를 `n_heads` 개수와 일치시킴으로써 모델이 여러 개의 독립적인 문맥 행렬을 생성할 수 있게 됩니다.
* 8M 모델 기준 새로운 전체 용량: `4개 헤드 * (72 x 72) = 20,736개 요소`.
* 이로써 파라미터당 표현력을 4배로 극대화하여 메모리(문맥)의 심각한 소실 없이 4096 길이의 데이터를 충실히 압축할 수 있습니다 (추론 속도는 토큰당 O(1)로 유지됨).
